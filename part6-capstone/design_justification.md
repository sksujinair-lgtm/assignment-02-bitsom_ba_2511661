## Storage Systems
Each of the four goals has distinct data access patterns, which drove the storage choices.
Goal 1 — Predict patient readmission risk uses a data warehouse (e.g., Amazon Redshift or BigQuery). Historical EHR records are large, structured, and queried in bulk for model training and batch scoring. A columnar warehouse handles these aggregations efficiently — a row-oriented transactional database would be far too slow for the workload required to train an XGBoost or LSTM model.
Goal 2 — Allow doctors to query patient history in plain English uses a vector database (e.g., Pinecone or pgvector). Clinical notes are embedded into high-dimensional vectors, and the RAG pipeline performs semantic similarity search to retrieve relevant records for a natural language query. A relational database cannot perform approximate nearest-neighbour search efficiently, making a vector store the right fit.
Goal 3 — Generate monthly reports uses the same data warehouse for aggregation queries, with an object store (e.g., AWS S3) for the final PDF or BI-ready report artefacts. Hospital operations data — bed occupancy, departmental costs — is summarised on a monthly cadence, which is exactly the analytical workload warehouses are designed for.
Goal 4 — Stream and store real-time vitals uses a time-series database (e.g., InfluxDB), fed via a Kafka message broker. ICU devices produce continuous, high-frequency, timestamped readings. Time-series databases are purpose-built for this pattern — they offer fast writes, efficient compression of sequential data, and built-in time-window queries that general-purpose databases handle poorly at scale.

## OLTP vs OLAP Boundary
The OLTP boundary sits at the EHR/EMR and hospital operations source systems. These handle day-to-day transactional work — registering patients, recording treatments, updating bed assignments — and are optimised for low-latency reads and writes on individual records with strong ACID guarantees.
The OLAP boundary begins at the batch ETL pipeline and data warehouse. Once data is extracted and loaded into the warehouse, it enters the analytical layer where queries are read-heavy, span large date ranges, and involve aggregations across many records. The ETL pipeline is the explicit handoff point, transforming row-oriented transactional data into columnar analytical structures. The real-time path (Kafka → Time-series DB) sits outside this classic divide, representing a stream-processing pattern that feeds the alert engine without touching the warehouse.

## Trade-offs
The most significant trade-off is data freshness vs. complexity. The nightly batch ETL means the readmission model scores patients on data that may be up to 24 hours old — a meaningful lag in a clinical context.
The mitigation is two-fold. First, the real-time streaming path handles ICU vitals independently, so the most time-critical use case is never batch-delayed.
